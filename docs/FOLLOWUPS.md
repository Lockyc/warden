# warden — deferred follow-ups

Known, intentionally-deferred work. Each item is a conscious deferral, not an oversight — recorded here so it isn't lost. Remove an item when it's done.

## To add in `warden-app` when needed

These are trivial, zero-retrofit additions to make the moment `warden-app` needs them; deferred per YAGNI.

- **Add `serde` derives to the model types** (`Config`, `Window`, `Tab`, `Warning`, `Colour`) — only if a future slice chooses IPC-direct model types over the app's own DTOs. `warden-app` deliberately maps `Window`/`Tab` → app DTOs (web contract decoupled from the model), so this stays deferred. Tauri IPC `#[command]` return types must be `Serialize` to cross to the web chrome. `serde` is already a dependency — add the `derive`/`serialize` feature and the derives when the IPC layer lands, rather than maintaining parallel DTOs.

## Watcher robustness (deferred to `warden-app`)

- **No debounce / coalescing** in `Watcher` (`crates/warden-config/src/watch.rs`). The callback fires for every filesystem event matching the config file name. Editors that write in place (rather than atomic temp-file + rename) can produce a transient `load()` parse error (a partial read mid-write) and/or multiple callbacks per save. Atomic-save editors are unaffected. Debounce/coalescing is intentionally left to `warden-app`, which owns the reload UX and already keeps last-good config on a parse error. Also documented at the call site.

## Lower-priority / edge cases

- **`warden validate` exit code does not distinguish "ok" from "ok with warnings"** (`crates/warden-config/src/bin/warden.rs`) — both exit 0. Not spec-mandated; revisit if a CI consumer needs to gate on nonexistent-dir warnings.
- **Soft degradation when `dirs::home_dir()` returns `None`** (HOME unset, e.g. some CI): `~/…` is left literal (later surfaces as a dir-missing warning) and `config_path()` falls back to a relative `.config/warden/config.toml`. Degenerate environments only.
- **Tilde expansion handles only `~/`** (`resolve.rs`), not bare `~` or `~user`. Within current spec scope (examples only use `~/…`).

## Test-isolation note (inert)

- `config_path_respects_env` (`crates/warden-config/src/load.rs`) mutates the process-global `WARDEN_CONFIG`. Cleanup is panic-safe (removed before the assert), but if a second test that reads `config_path()`/`WARDEN_CONFIG` is ever added, serialize them (e.g. `serial_test`) or refactor `config_path` to take an injected override — Rust runs tests in parallel threads. Inert today (no other reader).

## `warden-app` surface-embed hardening

`crates/warden-app` embeds libghostty terminal surfaces in Tauri on macOS. The following hardening is deferred, by priority:

- **libghostty is a throwaway prebuilt.** Vendored artifact is `Lakr233/libghostty-spm` `storage.1.2.7` — third-party, iOS-patched, ~Ghostty 1.2.7 — committed as a 39MB static `libghostty.a`. Replace with a controlled **upstream source build** (pinned commit). The source build is currently blocked on this machine: Ghostty pins **Zig 0.15.2**, which cannot link the **macOS 26.5 SDK** (undefined libSystem symbols incl. `__availability_version_check`, reproduces on a trivial hello-world). Revisit when a Zig that supports the macOS 26 SDK also builds the chosen Ghostty ref, or build on a CI runner with a compatible SDK. Also: don't commit the binary — use Git LFS or fetch-in-`build.rs`.
- **Make the `TerminalSurface` seam object-safe.** `close(self)` (by value) makes `Box<dyn TerminalSurface>` impossible, so the registry stores the concrete `GhosttySurface` — compile-time backend swap only, not runtime polymorphism (can't hold heterogeneous surfaces). Use `close(&mut self)` or Drop-based teardown. Pairs with: **add `Drop` for `GhosttySurface`** (today a surface freed only via explicit `close`/`close_all`; a dropped-without-close surface leaks the libghostty surface + its shell pty). Concrete motivation: any path that drops a `GhosttySurface` without calling `close` leaks the libghostty surface + its shell pty, so a `Drop` impl that frees is the durable guard (mind the double-free vs `close(self)` — Drop must become the sole free path). (The former panic-on-spawn-failure trigger is gone: `Registry::add`/`ensure_spawned` now return `Result` and a failed spawn leaves the tab cold — nothing half-created to leak — but Drop still removes the reliance on remembering to `close`.)
- **Finish `cfg(macos)` gating for an eventual Linux port.** The misleading-cross-platform-scaffolding problem is resolved: `surface/mod.rs:9` now gates `pub mod ghostty;` behind `cfg(target_os = "macos")`, and a non-macOS build fails honestly and immediately via `main.rs`'s `#[cfg(not(target_os = "macos"))] compile_error!(...)` rather than a confusing "can't find crate `objc2`". What remains for an *actual* Linux backend: `mod ffi;`/`mod surface;` in `main.rs` are still ungated (moot today behind the `compile_error!`), so when Linux work starts they (and the `ffi`/`objc2` deps) need gating alongside removing the `compile_error!`. Mechanical; do when Linux work starts.
- **`Registry::new` takes a raw `*mut c_void` NSWindow pointer** (`registry.rs`) — the registry interface is itself platform-aware, so a non-macOS backend can't implement it without knowing what the pointer means. Pass a platform-opaque handle or a surface-factory closure so the registry stays platform-agnostic (the seam's intent).
- **Move libghostty lifecycle init behind the seam.** `ffi::ghostty_init` is still called from `main.rs`; the surface layer should own libghostty init so `main.rs` never names libghostty (the seam constraint).
- **Single-source the sidebar width.** The default `160` is duplicated in three sites that must be kept in lockstep — `manager.rs` `INITIAL_RECT` (`x: 160.0`), `ui/index.html` CSS (`#sidebar width: 160px`), and `ui/index.html` JS (`SIDEBAR_DEFAULT_WIDTH = 160`). Collapse to one source so they can't drift. (The curator-style chrome look itself is done — this is the lone remaining width-duplication cleanup.)
- **Terminal input — remaining gaps.** Wired (`surface/ghostty.rs`): control-key encoding (`is_printable_text` keeps control chars out of `text` so legacy apps get bare control bytes, not Kitty `CSI…u`), ⌘-combos (`performKeyEquivalent:` forwards to libghostty's bindings — ⌘-presses arrive there, *not* `keyDown:`), mouse button/drag/scroll, and copy/paste (clipboard callbacks ↔ `NSPasteboard`, routed to the focused surface via `FOCUSED_SURFACE`). Still unhandled: **Alt/Meta** combos (Option yields the macOS composed char as text rather than an ESC-prefixed Meta sequence), **function/arrow/nav keys** (no explicit encoding — whatever libghostty derives from `keycode`), **IME / dead keys / marked text** (no `NSTextInputClient`), and **`mouseMoved`** hover reporting (needs an `NSTrackingArea`; click/drag/scroll work without it).
- **Full keybindings + ad-hoc tabs.** Tab navigation is wired as **menu items** (the macOS-native, focus-independent path): a **Tab** submenu with **Previous Tab ⌘⇧[ / Next Tab ⌘⇧]** and digit chords whose binding follows the global `tab_digit_keys` config (`"jump"` default → **⌘1–⌘9** jump-to-position; `"cycle"` → **⌘1 / ⌘2** next/prev + **⌘3–⌘9** jump), plus an app submenu with **Quit ⌘Q / Close ⌘W / Minimize ⌘M** (all in `main.rs` setup). Items fire the Builder's `on_menu_event`, which `emit_to`s the focused window (`warden:cycle-tab` ±1 / `warden:select-tab` n); the chrome reuses `select()`. We deliberately did **not** use ⌘+arrow — it collides with macOS line/document text-nav and could only be captured per-view (focus-dependent, fighting the OS); the menu is checked app-wide before any view. Still deferred: **`cmd+\`` to cycle *windows***, and **ad-hoc tabs/windows** (`cmd+T`/`cmd+N`).
- **FFI hardening.** C-populated `#[repr(C)] enum` fields in `ffi/mod.rs` (`platform_tag`/`backend`/`context`) risk invalid-discriminant UB if libghostty ever returns an unknown value — note the UB triggers the instant `ghostty_surface_config_new()` returns the struct **by value** (in `GhosttySurface::new`, `surface/ghostty.rs`) and Rust materializes those enum fields from C bytes, before any field is overwritten; use integer newtypes (as already done for `mods`). The `size_of` asserts guard total size, not field offsets.
- **Pin the "libghostty copies strings during the call" assumption.** Correctness of `ghostty_surface_new` (and `ghostty_app_new`) depends on libghostty copying the config strings (`working_directory`/`command`) and the runtime config *synchronously during the call* — they're stack locals (in `GhosttySurface::new`, `surface/ghostty.rs`) that die on return. Almost certainly true (matches Ghostty's macOS app), but unverified against the header and currently the single most consequential unchecked invariant: a future lib/header bump that retained the pointers would be a silent UAF. Verify against the ghostty source the prebuilt was cut from; consider a defensive comment/test at the call site.
- **Tighten the IPC boundary (CSP hardening).** A restrictive CSP **is now set** (`tauri.conf.json`): `default-src 'self'`, `object-src 'none'`, `frame-src 'none'`, `base-uri 'self'`, `img-src 'self' data:`, `connect-src 'self' ipc: http://ipc.localhost`. It still allows **`'unsafe-inline'`** for `script-src`/`style-src` because `index.html`'s `<script>`/`<style>` are inline — so the CSP blocks remote loads but not inline injection. Config colour still reaches the DOM (`index.html` sets the sidebar background from the window colour via `tintOverBase(windowColour, …)`), but only as a config-core `Colour`-validated hex composited into a `style.background` value, so it isn't an open sink today. Residual hardening: externalize or nonce the inline `<script>`/`<style>` and drop `'unsafe-inline'`, and revisit `withGlobalTauri: true` (the full IPC bridge is exposed to the renderer). `set_hole_rect`'s f64-rect validation is also done (`main.rs` rejects non-finite + clamps to ±100k before converting).
- **Packaging now signs + notarizes; a `.dmg` target is the residual.** Release `just build`/`just deploy` produce a `warden.app` that **is now signed with Developer ID and notarized + stapled** when the Apple signing/notary env vars are set (`APPLE_SIGNING_IDENTITY` pointing at a Developer ID Application cert, plus `APPLE_ID`/`APPLE_PASSWORD`/`APPLE_TEAM_ID`, or `APPLE_API_KEY*`); Tauri runs the signing and `xcrun notarytool` submission during `cargo tauri build`. With those vars unset the build is ad-hoc/unsigned and `scripts/install-app.sh` strips the Gatekeeper quarantine xattr so a local copy runs. Releases currently attach a zipped `.app`; the only residual nicety is a **`.dmg` bundle target** (a draggable installer vs a zip), add one if wanted. (The `macos-private-api` item below independently bars the Mac App Store route.)
- **`macos-private-api` is now enabled** (`Cargo.toml` / `tauri.conf.json`) because windows are built at runtime and `WebviewWindowBuilder::transparent` is gated behind that feature on macOS. This uses private AppKit API, so it **bars Mac App Store distribution** — fine for the current direct-distribution target, but revisit if MAS shipping is ever wanted (would need a non-private transparency route or an opaque-window design).
- **Diagnostic-window close leaves app running with zero windows — by design.** The diagnostic window (shown on a missing/invalid config at launch) has no `Destroyed→quit` handler; closing it keeps the app alive. This is intentional: the file watcher stays active and the next valid config save materializes the windows via the normal recovery path. A reopen-from-menu affordance is deferred with the rest of the menu/ad-hoc work.
- **Open windows on demand after close (window lifecycle).** Today, closing the last window of a config-defined window permanently removes it for the session — there is no way to reopen it without editing and reloading the config. A window lifecycle feature would let users open/reopen config-defined windows on demand. May want a config knob (e.g. `open_on_launch = false`) to control which windows auto-open at startup vs. appear only on request.
- **`open_on_launch` parity (after lifecycle lands).** Once the window lifecycle feature exists, warden should expose a per-window `open_on_launch` flag (analogous to the per-tab `load_on_open`) so users can mark windows that should always materialize at startup vs. those kept available but not opened automatically.
- **Decide ghostty argv passthrough.** `main.rs` (the `ghostty_init` block in `main`) forwards the full process `std::env::args()` into `ghostty_init`, so any ghostty CLI flag is injectable via warden's own argv. Decide: support passthrough (keep) vs pass a fixed `["warden"]` argv (recommended unless passthrough is a wanted feature). Also: those argv `CString` pointers are cast `*const`→`*mut c_char` for `char**` — verify ghostty treats argv as read-only, else hand it owned mutable buffers.
