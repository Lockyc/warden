# warden — agent orientation

## What warden is

warden is a **config-driven terminal multiplexer** — "curator for terminals." A single TOML file is the source of truth: it defines **windows** and the **project tabs** inside them. The app materializes itself from that config and **hot-reloads on save**. Each window carries a colour + title banner for at-a-glance identity; each tab is a real terminal opened in a working directory running an optional command. warden is **generic and content-agnostic** — it knows nothing about any specific tool; the command a tab runs is arbitrary (a shell, a TUI, an agent launcher, whatever).

Target platform: **macOS**. Linux is a *possible future consideration, not a guarantee* — the `warden-config` crate is kept platform-neutral (pure logic, no macOS-only APIs) so that door stays open, but nothing commits to shipping Linux. Not Windows.

## Current state — read this before assuming anything exists

Two layers are built:

- **Config-core** — the `warden-config` crate (pure logic, no GUI): parses/validates/resolves the TOML, diffs two configs for hot-reload, loads from disk, watches the file, ships a `warden validate` CLI.
- **GUI** — the `warden-app` crate: a config-driven, multi-window **macOS-only** Tauri app. It loads the config at launch and materializes **a native window for each `[[window]]`**, each with its colour + title banner; tabs are embedded libghostty terminal surfaces behind the `TerminalSurface` seam; each tab lazy-spawns on first focus, except `load_on_open = true` tabs which spawn at window open and keep running for background work. Chrome is curator-style: a transparent window with an Overlay titlebar (terminal reaches the top, traffic lights over the sidebar), an opaque draggable sidebar (banner + tab list). Tabs can be **grouped**: a `[[window.group]]` renders as a labelled sidebar section over its tabs, with loose (ungrouped) `[[window.tab]]`s in a headerless section first — purely presentational (see the config-schema section). Each tab row carries a **live/cold dot** (filled green = surface spawned; hollow ring = cold) that doubles as an **unload** control: hovering a live dot reveals a red ✕ that kills that terminal (surface + PTY) — the tab goes cold and respawns a fresh shell on next focus (the normal lazy-spawn path, *not* a curator-style cheap reload). Unloading the visible tab switches to an already-**live** neighbour (next if live, else the nearest live tab, leaning to the previous side) — keyboard navigation never wakes a cold tab, so if nothing else is live the hole shows a **blank-background placeholder** (warden's shield mark in muted grey, curator's empty-screen idiom) until you pick a tab. Tabs also surface **terminal notifications**: when a tab rings the bell or emits an OSC 9 / OSC 777 desktop-notification escape (decoded from libghostty's `action_cb` → a seam-neutral event → `notify.rs`), warden badges that tab's row with an amber dot when it isn't the visible tab, and a desktop-notification additionally raises a macOS banner; the badge clears on focus. **Clicking the banner surfaces its tab** — the banner carries the originating `(window label, tab id)` in its `userInfo`, and the notification-click delegate raises that window (activating warden from the background) and selects the tab (best-effort: a since-unloaded/removed tab still raises the window). Any program printing the standard escape benefits — this is the channel agentmux's Claude hooks feed (tmux passthrough + OSC 777) instead of shelling `osascript`. Probe-enabled tabs also carry a **session-presence dot** (cyan): warden runs each tab's `probe` command (via `sh -c`, cwd = tab dir) on a `probe_interval`-second timer and lights the dot when it exits 0 — independent of whether the terminal surface is loaded; the canonical use is an amux session check. **Hot-reload** is live (watcher + `reconcile` → per-window open/close/refresh; last-window-quit). A missing/invalid/empty config opens a **diagnostic window**; a parse error on a live save shows an **error banner** in the chrome and keeps last-good windows up; a recovered config materializes and closes the diagnostic. Each window's **size + position (+ maximized)** is persisted across launches per Tauri label via `tauri-plugin-window-state` (the `[[window]]`'s `inner_size` is just the first-run default); the per-window **sidebar width** is persisted separately in `localStorage` by the chrome.

Tab navigation is wired through the app menu: **⌘⇧[ / ⌘⇧]** cycle the previous/next tab but **only among loaded tabs** (cold tabs are skipped — cycling never spawns). The **digit chords are mode-selected by the global `tab_digit_keys` config**: `"jump"` (default) gives the standard **⌘1–⌘9** jump-to-position; `"cycle"` makes **⌘1 = next / ⌘2 = previous** (aliasing ⌘⇧] / ⌘⇧[) and shifts jumps to **⌘3–⌘9** (positions 1–2 then have no direct jump chord — the deliberate tradeoff). The menu (Tab + Config + Window submenus) is rebuilt wholesale on launch, whenever a hot-reload flips the `tab_digit_keys` mode, and on any window-set change — open, close, reopen (`rebuild_menu`/`build_app_menu` in `main.rs`, which must run with the `ManagerState` lock released — the mutex is non-reentrant); the `on_menu_event` handler is mode-agnostic (keys on item IDs). A **Config** submenu (**Edit Config** / **Reveal Config in Finder**, no accelerators — mirrors curator) runs `open` / `open -R` on `warden_config::config_path()`; like the Window items it acts on the app/config, so it's routed *before* the focused-window lookup in `on_menu_event`. **⌘W** unloads the active tab, **⌘⇧W** closes the window, plus **⌘Q/⌘M**. (Safari/Chrome convention: ⌘W = close *tab*, ⌘⇧W = close *window*; here close-tab means unload.) A **Window** menu lists every configured window: open ones carry a checkmark and raise on select, closed ones show **(closed)** and reopen on select (same label ⇒ its persisted size/position restores; tabs come back cold). **⌘⇧T** reopens the most-recently-closed window. Still deferred (see `docs/FOLLOWUPS.md`): `cmd+\`` window cycling and ad-hoc tabs (`cmd+T`/`cmd+N`); a real **libghostty source build** (the vendored `libghostty.a` is a throwaway prebuilt; the source build is blocked on Zig 0.15.2 vs the macOS 26 SDK); `TerminalSurface` seam hardening (object-safety, `Drop`-based teardown, platform-opaque window handle); FFI repr hardening; libghostty lifecycle behind the seam; IPC hardening (a restrictive CSP now ships — what remains is dropping `'unsafe-inline'` and reviewing `withGlobalTauri`); watcher debounce; argv passthrough decision.

## Intended architecture (where it's going)

- **Tauri** (Rust core + web chrome), one app, macOS-first. Tauri's cross-platform nature keeps a future Linux port viable, but macOS is the only committed target.
- **Windows = native macOS windows** — each `[[window]]` in config is one window (macOS `cmd+\`` cycles them), with its own colour + title banner. **Tabs = projects** within a window, cycled by a secondary keybinding. A single app/dock-icon by design — per-window identity is carried in-app via the banner, not via separate app bundles.
- **Terminal surface = embedded libghostty** behind a `TerminalSurface` seam — a small per-OS native shim hosts the surface (NSView on macOS; a GTK widget would host it on Linux if that port ever happens). A non-libghostty fallback impl (e.g. SwiftTerm-class) can sit behind the same seam. libghostty's embedding C API is officially unstable — **pin it** to a known commit.
- The `warden-config` crate is the foundation the app consumes (window/tab set, hot-reload reconcile, watcher).

## Workspace layout

Cargo workspace. `crates/warden-config` — the config crate (a library plus the `warden` CLI binary). Data flows one direction through small, independently-tested modules:

```
raw.rs       serde structs mirroring the TOML schema + parse()
  ↓
resolve.rs   validate + fill defaults + expand ~ → (Config, Vec<Warning>); ResolveError
  ↓
model.rs     clean resolved types: Config / Window / Tab / Warning
reconcile.rs reconcile(old, new) → Reconciliation (open/close windows, per-window colour/tab-add-remove/tab-reorder) for hot-reload
load.rs      load(path) = read+parse+resolve; config_path() (WARDEN_CONFIG override else ~/.config/warden/config.toml); LoadError
watch.rs     Watcher — notify-based parent-dir file watcher; fires load() on change
(fmt + colour live in the shared config-core crate — re-exported at warden-config's root as format_str/format_file + Colour/ColourError; see Config formatting footgun)
bin/warden.rs  `warden validate [path]` + `warden fmt [--check] [path]` CLI
```

`crates/warden-app` — the macOS Tauri app that consumes the config crate. Key modules: `plan.rs` (config → `WindowSpec`/`TabPlan`, `reconcile` → `WindowOp`), `manager.rs` (`WindowManager`: materialize windows, apply reconciliations, diagnostic window), `registry.rs` (per-window tab registry over surfaces), `surface/` + `ffi/` (the libghostty embed behind the `TerminalSurface` seam — macOS/objc2), `geometry.rs` (web-rect ↔ NSView-rect). `ui/index.html` is the chrome; `ui/diagnostic.html` the config-error page.

## Config schema (`~/.config/warden/config.toml`; override with `WARDEN_CONFIG`)

```toml
shell = "fish -l"                      # optional; global default shell every tab spawns (default: your login shell, run as login)
cmd   = "amux"                         # optional; global default startup command run inside the shell
format_on_save = true                  # optional; default false — rewrite this file tidy on each clean hot-reload
tab_digit_keys = "jump"                # optional; whole-app ⌘1/⌘2 mode: "jump" (default, ⌘1–9 jump)
                                       #   or "cycle" (⌘1 next / ⌘2 prev; jumps shift to ⌘3–9)
density = "comfortable"                # optional; whole-app chrome sizing: "comfortable" (default)
                                       #   or "compact" (proportionally condensed type + spacing)
probe = "…"                            # optional; session-presence probe, cascades global→window→tab (cyan dot on exit 0)
probe_interval = 5                     # optional; seconds between probe passes (default 5; 0 = focus/refresh-only)
kill  = "…"                            # optional; session-kill command, cascades global→window→tab; run fire-and-forget on confirm via the cyan dot
notify_debug = false                   # optional; default false — trace the notification path to /tmp/warden-notify-dbg.log (debug aid, read at launch; see docs/notifications.md)

[[window]]                             # = a native macOS window
title  = "work"                        # required, unique, non-empty; banner + window title
colour = "#0f8a8a"                     # optional; #rgb or #rrggbb; omit → neutral default
width  = 1500                          # optional; initial window width (px; default 1500)
height = 1000                          # optional; initial window height (px; default 1000)
shell  = "zsh"                         # optional; per-window shell override
cmd    = "amux"                        # optional; per-window startup override
probe  = "…"                           # optional; session-presence probe override for this window
kill   = "…"                           # optional; session-kill command override for this window

  [[window.tab]]                       # = a project terminal (loose/ungrouped → headerless section, first)
  title      = "alpha"                 # optional; default = basename(dir); non-empty & unique within the window
  dir        = "~/Developer/…/alpha"   # required
  shell      = "bash"                  # optional; per-tab shell override
  cmd        = "amux"                  # optional; per-tab startup override ("" = opt out → bare shell)
  load_on_open = true                  # optional; default false (spawn at launch + keep running for background work)
  probe      = "…"                     # optional; session-presence probe override for this tab ("" = opt out)
  kill       = "…"                     # optional; session-kill command override for this tab ("" = opt out)

  [[window.group]]                     # = a sidebar section (optional; loose tabs need no group)
  name = "backend"                     # required, non-empty, unique within the window

    [[window.group.tab]]               # same fields as [[window.tab]]; rendered under the group header
    dir = "~/Developer/…/api"
```

**`shell` and `cmd` cascade global → window → tab; the nearest set level wins.** A missing value at a level inherits from above; `shell` falls back to **your login shell** (`$SHELL`, run as a login shell — detected in `warden-app`/the CLI and injected via `resolve_with`, since the crate stays env-free; `DEFAULT_SHELL` is only the test/last-resort fallback) if unset everywhere; `cmd` is `None` (bare shell) if unset everywhere. An explicitly-empty `cmd = ""` counts as *set* and resets to `None`, so it opts a level out of an inherited command instead of inheriting it (see `cascade()` in `resolve.rs`). `shell` is treated the same (empty = unset). Resolution collapses the cascade into the flat `Tab.shell: String` + `Tab.startup: Option<String>` — the app never sees the levels.

**`probe` is a generic session-presence check.** It cascades global → window → tab exactly like `shell`/`cmd` (`""` opts a level out); resolution collapses to `Tab.probe: Option<String>`. warden runs it per tab via `sh -c` with cwd = the tab's dir on a timer (`probe_interval` seconds; `0` = focus/refresh-only), substituting `{dir}`/`{title}`, and **exit 0 ⇒ a session exists** → a cyan dot on the tab row. The crate/app stay tmux/amux-agnostic — the command is opaque; the canonical amux case is `"$HOME/.agentmux/bin/amux" --probe` (exit 0 when the dir's session exists — the agent on tmux's default socket **OR** a lingering frame on the `agentmux-frame` socket). amux owns the session naming + socket layout, so the probe can't drift from it — the inverse of hand-inlining `tmux has-session` legs in warden config, which break silently when amux changes a socket name or its name sanitisation. The frame leg lights the dot when an agent has exited but its frame wrapper is still up — the state `amux --kill` reaps (no separate term check: a frame's scratch terminal never outlives the frame); a plain bare-amux (no frame) only ever has the agent session. Name amux by **absolute path** (a Finder launch's minimal PATH won't find a bare `amux`; see the probe-exec-environment footgun). Independent of the live/cold dot (a cold tab can have a running session).

**`kill` is a companion session-kill command.** It cascades and resolves identically to `probe` (`""` opts a level out → `Tab.kill: Option<String>`). warden runs it **fire-and-forget** via `sh -c` (cwd = tab dir; `{dir}`/`{title}` substituted raw — quote them if paths may contain spaces) when the user confirms via a two-step click on the cyan presence dot. Killing severs the session **without unloading warden's terminal surface** — the surface stays open at a live shell prompt after the session is gone; warden then re-runs the probe to update the dot. The arm-state (the pending confirm) lives in the chrome only — no IPC until confirm. Because the kill control lives **on** the presence dot, `kill` is only reachable on tabs that also set `probe` — no `probe` ⇒ no cyan dot ⇒ no way to trigger the kill (the command still resolves, it just renders no affordance). This coupling is by design (documented, not enforced): a tab inheriting a global `kill` while opting out of `probe` has a silently inert kill. `kill` inherits the same probe PATH/env footgun (now half-closed by `restore_login_path` — see the probe-env footgun): warden imports the login-shell PATH at startup so a packaged `warden.app` resolves bare Homebrew binaries, but absolute paths stay the robust fallback — warden imports PATH only, not your shell exports. The canonical kill is `"$HOME/.agentmux/bin/amux" --kill`, the mirror of `amux --probe`: with cwd = the tab dir it tears down the **whole project** — the agent session on the default socket plus its `-frame`/`-term` sessions on the dedicated sockets — so it reaps exactly what the probe detects. (`amux --kill` itself shells bare `tmux`; same login-PATH resolution.) The example config delegates to it rather than a raw `tmux kill-session`, which would kill only the agent and orphan the frame/term.

**`cmd` runs inside the shell, it doesn't replace it.** Every tab spawns its resolved `shell` (an interactive shell under the PTY); the resolved `cmd`, if any, is delivered as libghostty `initial_input` — i.e. *typed into* that shell (newline-terminated) rather than exec'd directly. This is deliberate and load-bearing: `amux` is a shell **function**, not an executable, so execing it directly fails — only an interactive shell resolves it. As a bonus the shell stays live after the command exits (detaching from `amux`/tmux drops you to a prompt, not a dead pane). Do **not** "fix" this back to passing `cmd` as libghostty's `command`/exec target.

**Groups are presentation only — a sidebar sectioning concern, not a new container of sessions.** `[[window.group]]` blocks hold `[[window.group.tab]]`s; resolution **flattens** loose tabs + every group's tabs into the one ordered `Tab` list (loose first, then groups in file order), tagging each with `Tab.group: Option<String>` (`None` = loose). Crucially, **the flat `Tab` list is the only downstream shape** — `reconcile`, the registry, tab navigation (⌘1–9 / cycle), the live/cold dot, unload, and notifications all stay flat and key-based; only the chrome re-sections by `group` at render time. Groups add **no cascade level** (a grouped tab cascades tab → window → global exactly like a loose one). A kept tab's `group`/`probe`/`kill` change **is** detected by `reconcile` (via `WindowUpdate.set_meta`) and applied live — sidebar re-section for `group`, new probe/kill commands picked up on the next poll/kill — without respawning the PTY. These are the deliberate exceptions to the "in-place tab field edits aren't detected" rule: grouping is cosmetic and must not cost a terminal its session; probe/kill are externally-run commands whose only use of the app is the dot and kill trigger, so they too update live. This nested-authoring → flat-label shape matches how curator itself models groups (it flattens `[[group]]` to a per-tab label too).

**`density` is a whole-app chrome-sizing mode** — `"comfortable"` (default) or `"compact"`. It resolves to `Config.density: Density` (global, no cascade; unknown value → `ResolveError`, like `tab_digit_keys`) and is carried per-window in the chrome DTO (`InitDto.density`, on init and hot-reload refresh — `apply()` takes the new density so a live flip restyles without a relaunch). warden resolves `density` and passes it in the DTO; **chrome-core** applies `data-density` on `<html>`, swapping its `--cc-*` sizing tokens (`--cc-row-font`, `--cc-tile-size`, `--cc-dot-size`, `--cc-sidebar-w`, paddings/gaps); `compact` is a proportional ~0.85× scale of the comfortable set. Tweak the scale in one place — chrome-core's `assets/sidebar.css` (`:root` / `[data-density="compact"]`); the sidebar's default + double-press-reset width reads `--cc-sidebar-w` so it follows density too. **Both apps consume chrome-core**, so the tokens live once and stay aligned by construction.

Validation: unique window title, unique tab title within a window (window-wide — across loose tabs and all groups), non-empty title/dir/explicit-title, non-empty + unique group name within a window, malformed colour → **errors**; width/height ≤ 0 → **error**; a `dir` that doesn't exist → **warning** (tab still created). Invalid config must be reported, never panic.

## Build / test / run

- `cargo build` / `cargo test` (workspace). `warden-app` is macOS-only (libghostty embed); it fails to compile elsewhere by design.
- **Toolchain is pinned** in `rust-toolchain.toml` (currently `1.96.0`, with `rustfmt`+`clippy`) so formatting and lints are identical for everyone — an unpinned `stable` drifts and re-wraps committed code, which has bitten `just fmt`/`just check` here. rustup auto-installs the pinned version. **Keep it in lockstep with the shared crates — config-core and chrome-core — and with curator**: both are *git dependencies* (chrome-core a *build*-dependency), so their own pins don't apply when warden builds them — warden's toolchain compiles the whole graph. Bump all of them together (curator's toolchain, config-core's rev, chrome-core's rev), naturally when bumping a shared crate's pinned rev.
- `cargo run -p warden-config --bin warden -- validate [path]` — validates a config and prints the resolved window/tab tree + warnings (exit 0 ok / 1 load error / 2 usage).
- `cargo run -p warden-config --bin warden -- fmt [--check] [path]` — formats a config in the shared house style (config-core's taplo-backed formatter — no external taplo binary needed). `--check` is non-mutating and exits 1 if the file would change (for a pre-merge gate).
- `cargo run -p warden-app` — launch the GUI (reads `WARDEN_CONFIG` or `~/.config/warden/config.toml`).
- **`justfile`** wraps the common flows: `just run` (launches the app against `examples/config.toml` via `WARDEN_CONFIG`, so iterating never touches your real config), `just validate [path]`, `just test`, `just check`, `just fmt`, `just clippy`; bare `just` lists them.
- **Packaging:** `just build` produces the release `warden.app` (needs the Tauri CLI — `cargo install tauri-cli --version ^2`); `just deploy` (now via `install.sh`) installs it to `/Applications` and relaunches. Release builds are **signed with Developer ID and notarized + stapled automatically** when the Apple signing/notary env vars are present in the build environment — `APPLE_SIGNING_IDENTITY` pointing at a Developer ID Application cert, plus `APPLE_ID`/`APPLE_PASSWORD`/`APPLE_TEAM_ID` (or `APPLE_API_KEY`/`APPLE_API_ISSUER`/`APPLE_API_KEY_PATH`); Tauri runs the signing and `notarytool` submission (with hardened runtime) during `cargo tauri build`. With those vars unset (e.g. a contributor building from source), the build falls back to an ad-hoc/unsigned bundle, and `scripts/install-app.sh` strips the Gatekeeper quarantine xattr so a local copy runs either way. This is env-driven, not pinned in `tauri.conf.json`, so a build environment without the vars silently ships unsigned — set them to ship a distributable release. The bundle icon is `crates/warden-app/icons/icon.icns`, produced from the SVG masters by `assets/build-icons.sh`.
- **Releases:** the version is **single-sourced in `crates/warden-app/Cargo.toml`** — `tauri.conf.json` deliberately has **no `version` key** so the bundle inherits Cargo.toml's (don't re-add it; two version fields silently drift and the `.app` ships mislabelled). `warden-config` keeps its own independent version; the release tag tracks `warden-app`. Cutting a release (`main` tracks the latest release and is a clean ancestor of `dev`): on `dev`, bump `warden-app`'s version + refresh `Cargo.lock` → `chore(release): warden <v>` commit → `git tag v<v>` → fast-forward `main` to it → push `dev`, `main`, the tag → `gh release create v<v>` with notes summarising what shipped since the previous tag → **`just release`** (= `scripts/release.sh`) to build the notarized `warden.app`, `ditto`-zip it to `warden-<v>-macos.zip`, and attach it to the release. `scripts/release.sh` reads the version from Cargo.toml and **refuses to run unsigned** (no `APPLE_SIGNING_IDENTITY` → exit) so an un-notarized zip can't masquerade as a release artifact — every published release carries a notarized, Gatekeeper-clean `.app`.
- **Install from source:** `install.sh` (repo root) is the self-sufficient installer — `bash install.sh` from a checkout builds the current tree; `curl -fsSL https://raw.githubusercontent.com/lockyc/warden/main/install.sh | bash` clones/updates `~/.warden` and builds from it. Both backstop the Tauri CLI (`cargo install tauri-cli`), build via `cargo tauri build`, install to `/Applications/warden.app` (reusing `scripts/install-app.sh`), and seed `~/.config/warden/config.toml` if absent. `just deploy` delegates to `install.sh` (IN_REPO mode) then relaunches. The guided path is the **`/warden:install`** Claude command (`.claude/commands/warden/install.md`), which adds prerequisite checks and a config-edit/launch flow.

## Conventions & footguns (things that have bitten us / will bite again)

- **macOS app-icon safe area (the "our icon is bigger than everyone else's" bug):** Apple's icon grid sits the rounded tile in an **824×824 box centred on the 1024 canvas (~100px transparent margin each side)**. An edge-to-edge tile renders ~25% oversized in the Dock / cmd-Tab switcher next to every other app. `assets/icon-app.svg` deliberately draws the tile **edge-to-edge** (design it full-bleed); the margin is enforced in `assets/build-icons.sh`, which embeds the tile as a data: URI into an 824/1024 wrapper before rasterising — so the margin can't silently regress and the art never has to encode it. Do **not** "fix" this by making the SVG full-canvas again, and do **not** drop the wrapper. (This class of bug has recurred on every app; the script-level enforcement is the durable fix.)

- **Raw-string test fixtures with a colour hex:** a Rust `r#"..."#` literal containing `colour = "#..."` self-terminates early at the `"#`. Use `r##"..."##` for any TOML test fixture that includes a hex colour. (Hit repeatedly across modules.)
- **The watcher matches the config file by file name, not full path** (`watch.rs`) — deliberately. This survives macOS FSEvents reporting `/private/var/...` vs a caller's `/var/...` symlink, and survives atomic-save editors that swap the file inode (the watch is on the parent dir). Do **not** "fix" it back to full-path equality, and do **not** add an `event.kind` filter — spec requires firing on *any* event for the file (atomic-save renames can surface as `Create`).
- **Format-on-save rewrites the user's config but can't loop the watcher** (config-core's `format_file` + `main.rs` reload hook): when `format_on_save` is set, a clean reload calls `format_file`, which rewrites the file **only if the formatted bytes differ** (diff-guard) and **atomically** (temp file in the same dir + rename). warden's own write retriggers the watcher once, but the now-formatted file is byte-identical → no write → it settles. Two invariants hold this together — don't break them: formatting must stay **idempotent** (so the guard converges), and the hook runs **only on the clean-parse branch** (never rewrite a mid-edit-broken file; that path keeps last-good + shows the error banner). The house style lives in config-core's `format_str` (taplo `indent_tables`/`indent_entries`/`align_entries`/`align_comments`, `reorder_keys=false`, `allowed_blank_lines=1`, LF + trailing newline) plus a `separate_sections` post-pass that puts **one blank line before each container section** (`[[window]]`, `[[window.group]]`) while keeping **nested tabs tight** (no blank before a `[[…tab]]`) — taplo can only *cap* blank lines, so the pass both inserts the container separators and strips blanks that crept in before a tab. Tight-vs-separated keys on the header's leaf segment (`tab` = tight). A comment glued to a header counts as part of that section, so the inserted/stripped blank applies above the comment. config-core's formatter is the single source of truth for both the app hook and `warden fmt`, and its tests keep `separate_sections` **idempotent** (the second pass must be a no-op); the warden-side invariant is just that the reload hook runs **only on the clean-parse branch**, so the diff-guard converges.
- **Tab identity is the resolved title** (`Tab::key`), unique window-wide (across loose + grouped tabs). `reconcile` diffs windows by `title` and tabs by this key. In-place edits to a kept tab's `dir`/`cmd`/`shell`/`load_on_open` (title unchanged) are **not** detected — `warden-app` reopens the tab to apply them (so changing `dir` or `cmd` on a kept tab doesn't take effect until the tab is recreated or the app restarts). **`group`/`probe`/`kill` are the exception**: a kept tab's change to any of these IS detected (`WindowUpdate.set_meta`) and applied live without respawning — sidebar re-section for `group`, and the new `probe`/`kill` take effect on the next poll/kill, no restart. A window **title change** is destructive: it appears as close(old)+open(new), killing and recreating that window's terminals (including `load_on_open` ones).
- **Window size/position restore is per-label and runtime-triggered** (`tauri-plugin-window-state`): the plugin saves automatically on close/exit, but its *automatic* restore only fires for windows declared in `tauri.conf.json` — warden builds every window at runtime, so `manager.rs::build_window` must call `window.restore_state(SIZE|POSITION|MAXIMIZED)` itself right after `.build()`. State is keyed by the Tauri **label**, which `sanitize_label` derives deterministically from the window title (stable across launches → restore matches the right window). A window **title change** changes the label, so it restores fresh default bounds — consistent with a title change already being destructive (it recreates the window + terminals). The transient diagnostic window is excluded via `.skip_initial_state(DIAG_LABEL)` so its throwaway bounds never seed a real window. Don't drop the explicit `restore_state` call expecting the plugin to auto-restore. The builder sets first-run bounds via `.inner_size(spec.width, spec.height)` (config-resolved per-window `width`/`height`, defaulting to 1500×1000); the plugin overrides those with persisted bounds only when saved state exists for the label.
- **Generic:** keep tool-specific concepts out of the crate. Example command strings in tests stay neutral. The line between crate and public face: the **crate stays neutral**, but the **README/public face deliberately references [agentmux](https://github.com/lockyc/agentmux) (`amux`)** — a tmux-based agent launcher — as the canonical companion, since pairing warden tabs with it is the author's intended flow (the `warden → agentmux → tmux` stack, warden embedding libghostty as its terminal surfaces). Do not "degenericise" the README by stripping that link, and do not leak agentmux into the crate to match it.
- **warden scrubs `$TMUX`/`$TMUX_PANE` from spawned terminals' env** (`main.rs` `scrub_inherited_tmux_env`, called first thing in `main()`). warden is routinely launched from inside the very agentmux/tmux session it exists to host, and libghostty hands each surface's shell warden-app's own environment — so a leaked `$TMUX` makes nested tmux/agentmux think they're inside another tmux and refuse to open frames / misroute prefix keys. Don't remove the scrub, and if env passthrough is ever added to the surface spawn, keep these vars stripped (extend the list if other launcher-context vars cause the same nesting confusion).
- **App-menu shortcuts vs libghostty keybinds (the "tab hotkey only sometimes works" bug):** macOS sends a ⌘-combo to the key window's view hierarchy `performKeyEquivalent:` **before** the main menu, and libghostty has its *own* built-in keybinds for the standard tab chords (`⌘⇧[`/`⌘⇧]` = prev/next tab, `⌘1–9` = goto-tab — chords warden's Tab menu also binds; under `tab_digit_keys = "cycle"` warden instead maps `⌘1`/`⌘2` to next/prev and `⌘3–9` to jump). So if `WardenHostView::performKeyEquivalent:` forwards a chord to libghostty first, libghostty matches its keybind, returns *consumed = true*, the view returns `YES`, and the event is swallowed before warden's menu item ever fires. The fix (in `surface/ghostty.rs`): give the **main menu first refusal** — `NSApplication::sharedApplication().mainMenu().performKeyEquivalent(event)` at the top of `performKeyEquivalent:`; if it returns true the menu owns the chord, so stop. Only forward to libghostty when the menu declines. The menu's own accelerators define the reserved set, so it self-maintains — but don't "simplify" by forwarding to libghostty first, and remember any new menu accelerator automatically wins over a colliding ghostty keybind (that's intended).
- **libghostty cursor focus must track AppKit's first responder (the "cursor isn't filled but still types" bug):** libghostty draws a *filled* block cursor only while it believes the surface is focused (`ghostty_surface_set_focus(surface, true)`), a *hollow* one otherwise. Focus was pushed in only two paths — `GhosttySurface::focus()` (tab activate, via `makeFirstResponder`) and `performKeyEquivalent:` (⌘-chords, which sets the previous surface false). A plain **mouse-click into a terminal** makes the view first responder via AppKit directly, so `keyDown:` routes there (typing works) but nothing told libghostty — if another surface had grabbed focus via a chord, this one stayed false and rendered a hollow cursor while accepting input. Fix (in `surface/ghostty.rs`): a `becomeFirstResponder` override pushes `ghostty_surface_set_focus(true)` + retargets `FOCUSED_SURFACE` — that's the AppKit hook every first-responder *change* funnels through (click, `makeFirstResponder`), so the cursor tracks the real first responder. Don't drop it expecting `focus()`/`performKeyEquivalent:` to cover the click path — they don't. `becomeFirstResponder:` does **not** fire when a window merely re-becomes key with its old first responder intact, so it alone left the inverse glitch (filled cursor in a *non-key* window, or stale-hollow after the window returns). That's closed by per-window **`windowDidBecomeKey:`/`windowDidResignKey:`** observers (registered in `new`, scoped `object: window`, removed in `close`) that flip the active tab's focus flag with key state — gated on `!isHidden()` so only the keyed window's visible surface reacts (every tab view hears the notification; one is unhidden). The two mechanisms are complementary: first-responder for *which surface within a window*, key-state for *which window* — don't drop either.
- **The render layer's `contentsScale` must track the backing scale (the "terminal is scaled wrong after unplugging a monitor" bug):** libghostty's macOS renderer makes warden's `WardenHostView` **layer-hosting** — it assigns its own render layer to the view's `layer` and sets that layer's **`contentsScale` only at creation** (`renderer/Metal.zig`). Its live content-scale path (`ghostty_surface_set_content_scale` → `contentScaleCallback`) reflows font/cell metrics + framebuffer but **never touches the layer's `contentsScale`**. So when the window moves to a different-DPI display, Core Animation keeps compositing the layer at the *stale* scale and magnifies/shrinks the whole surface — even though the font itself reflowed. **The fix** (in the `viewDidChangeBackingProperties` override, `surface/ghostty.rs`): push `window.backingScaleFactor()` onto `self.layer`'s `contentsScale` ourselves, inside a `CATransaction` with actions disabled (so CA doesn't animate the change) — **exactly what Ghostty.app's own `SurfaceView` does** (confirmed by reading its AppKit surface view + `Metal.zig`). Session-preserving (no surface restart), runs unconditionally, and **verified on hardware** (unplug → re-renders in place). Still push content-scale + size together via `apply_surface_geometry()` (font reflow + framebuffer) — the `contentsScale` line is purely the *compositor's* scale, orthogonal to those. **If a DPI change ever looks wrong again, the lever is the layer scale — not the font path (it does reflow) and not respawning the surface (a removed dead end).**

- **A surface that fails to spawn degrades to a cold tab — never `.expect()`/`unwrap()` it (the "one bad tab crashes the whole app at launch" bug):** `GhosttySurface::new` returns `Result`, and `Registry::add` / `ensure_spawned` / `activate` propagate it (`registry.rs`). A failed spawn (null libghostty surface, interior-NUL in a config `dir`/`shell`) leaves the tab **cold** — it still shows in the sidebar and retries on next focus — and the reason is surfaced in the chrome error banner. Two channels by timing: a **launch/build-time** failure rides the **init snapshot** (`InitDto.error` → `manager.rs::build_window`), because the webview hasn't registered its `warden:error` listener yet so a pushed event would be lost; a **click-time** failure (`activate_tab`) is **pushed** as `warden:error` (chrome is listening). A hot-reload add-failure logs + retries-on-focus (no banner). Do **not** "simplify" any of these back to `.expect()`/`unwrap()`: the eager first-tab spawn runs at launch inside Tauri `setup`, so a single surface failure used to panic the *entire app* out of startup with no error shown.
- **A new *global* config setting won't hot-reload if you only thread it through the per-window DTO (the "density-only edit does nothing" bug):** `reconcile` diffs *windows/tabs*, and `apply()` emits `warden:refresh` **only for windows with a per-window diff** (colour / tab add-remove / order / `set_meta`). A global setting changed *alone* — `density`, like the menu's `tab_digit_keys` — produces an **empty reconcile**, so no refresh is emitted and the chrome never sees it. Wiring the global into `InitDto` *looks* sufficient because **launch works** (the webview pulls `init_dto`), which is exactly the trap: only the hot-reload path is broken. Each global needs its **own old-vs-new compare in `main.rs`'s reload hook**, beside the existing ones — `tab_digit_keys` rebuilds the app menu, `probe_interval` sets the shared atomic, and `density` calls `manager.refresh_all_chrome` (re-pushes every window's `init_dto`, which carries the now-advanced `last_good.density`). Add the explicit global-change branch; don't assume the per-window refresh covers it.
- **`emit_to` leaks to sibling webviews — per-window events must carry a `label` and the chrome must filter on it (the "both windows respond to one hotkey" bug):** with the global `listen` (`window.__TAURI__.event.listen`), `app.emit_to(label, …)` does **not** restrict delivery to that one window's webview — every window's listener receives the event. So every per-window emit stamps the focused window's `label` into its payload and the chrome drops events whose `label` ≠ its own (`myLabel`, learned from the init DTO): `warden:refresh` (manager.rs), the menu-driven `warden:cycle-tab` / `warden:select-tab` / `warden:unload-tab` (main.rs `on_menu_event` → `index.html`'s `forMe()` guard), `warden:notify` (notify.rs, the tab-badge event), and `warden:session-state` (probe.rs, the session-presence dot) all do this. Don't add a new per-window emit that relies on `emit_to`'s targeting alone — stamp the label and filter, or it fires in all windows. (A per-webview `getCurrentWebviewWindow().listen()` would scope on the receive side instead, but the project standard is the label stamp + filter for consistency with `warden:refresh`.)
- **Session-presence probes run off the main thread and must stamp the window label.** The probe poll loop (`probe.rs::run_pass`) snapshots each window's probe work-list under the `ManagerState` lock, then **releases the lock before** running `sh -c` (probes are slow; never hold the mutex across them). Results emit per-window via `warden:session-state` stamped with the window `label` and filtered by the chrome's `forMe()` — the same `emit_to`-leaks footgun as `warden:refresh`. `probe_interval` is shared as an `Arc<AtomicU64>` so a hot-reload changes cadence live; the poll loop waits in ≤3s slices and bails the moment the atomic changes, so even a large interval shortened to a small one (or `0`↔N) takes effect within a few seconds rather than after the old sleep. `0` disables the timer (focus + hot-reload still probe). Each window also probes itself once on load via the `probe_now` command, invoked by the chrome **only after** its `warden:session-state` listener is registered (`await sessionListenReady` in `index.html`'s `init()`) — so the first emit can't be lost to the listener-registration race. This per-window pull, not a launch-time push, is what populates the dots at startup; it's the only startup path when `probe_interval = 0` and it covers background windows that never get a launch `Focused`. Don't reintroduce a launch-time `spawn_pass(None)` to "populate early" — it races the listener and the dot can stay hollow. Probe `exit 0 = session present`; cwd = the tab's dir; tokens `{dir}`/`{title}` are substituted **raw** (not shell-quoted), so quote them in the command (`'… "{dir}"'`) when a path/title may contain spaces or `sh` metacharacters — otherwise the probe word-splits and silently reports "no session"; stdout/stderr are discarded so a chatty probe can't spam warden. Probes run **sequentially and without a timeout** on the poll thread (never the UI thread), so a probe that blocks stalls only the dot refresh — keep probe commands fast and non-blocking. Keep warden tmux/amux-agnostic — the command is the user's, warden only reads its exit code.
- **The default shell is your login shell (absolute, run as login) — warden is a terminal, so it behaves like one (`warden-app`'s `login_shell()`).** When no `shell` is set at any cascade level, warden defaults to `$SHELL` (fallback `/bin/zsh`) with `-l` appended — read in `warden-app`/the CLI binary (NOT the pure `warden-config` crate, which takes the default as data via `resolve_with`/`load_with`/`Watcher::with_default`; the bare `resolve`/`load` keep `DEFAULT_SHELL` for tests). `$SHELL` is an **absolute path** (launchd sets it from the user's directory record even on a Dock/Finder launch), so libghostty finds the shell without any PATH lookup — sidestepping the GUI-launch minimal-PATH trap *by construction*, the way Terminal.app/iTerm/Ghostty do — and `-l` makes it a login shell that sources config and builds PATH for the session. A config `shell` overrides it; warden being generic, an override is an arbitrary command. A **bare-name override** (`fish -l`) is *not* stuck on the minimal launchd PATH — `restore_login_path` (next footgun) imports the login-shell PATH at startup, so a bare Homebrew/nix shell resolves even from a Finder/Dock launch; an absolute path stays the robust fallback for a binary on an interactive-only PATH that `-l` doesn't source.
- **A GUI launch hides your PATH — warden imports the login-shell PATH at startup (`main.rs::restore_login_path`):** a `.app` opened from Dock/Finder/Spotlight inherits only the minimal launchd PATH (`/usr/bin:/bin:/usr/sbin:/sbin`), so a command named by **bare command** — a config `shell = "fish -l"`, a bare `tmux` in a `probe` — is `exec: … not found` and silently fails (the shell dies on spawn, symptom: a tab that only ever shows "Process exited. Press any key…"; a probe reports "no session"). The built-in *default* shell sidesteps this by construction (absolute `$SHELL -l`, previous footgun), but config-supplied commands are arbitrary and routinely bare. Rather than guess install prefixes (Homebrew/nix/MacPorts/custom all differ and a user may have none), `restore_login_path` runs the user's login shell once — `$SHELL -l -c`, reading PATH back via `printenv` (not `echo $PATH`, which is shell-syntax-dependent: fish space-joins list vars) between sentinels so rc-file chatter can't corrupt it — and adopts the PATH it builds, before any surface/probe spawns and inherits this process's env. Best-effort and self-limiting: a spawn failure, the 3s deadline (against a slow conda/nvm rc), or unparseable output leaves PATH exactly as inherited — never *worse* than the no-op. Caveat: `-l` captures the **login** env only, so a PATH exported solely in interactive-only rc isn't seen — an absolute-path shell/probe stays the robust fallback. Same lever as the tmux scrub: set warden-app's process env once and every surface/probe/kill inherits it. **Don't remove it again** expecting the absolute-`$SHELL` default to cover everything — it covers only the *unset* shell, not a config override or a probe/kill command (removing it once already regressed bare `fish`/`tmux` from a Finder launch).
- **Probes inherit warden-app's own process environment — the "probe works in `just run`, dead in the deployed app" trap, now half-closed by `restore_login_path` (above).** `probe.rs` runs the command as bare `Command::new("sh").arg("-c")` with no per-call PATH/env, so the probe sees whatever environment warden-app carries. **PATH is handled**: warden imports the login-shell PATH at startup, so a Finder/Dock launch no longer leaves it on the bare launchd PATH — a bare `tmux` resolves just as it does under `just run`. **Arbitrary shell *exports* are not** — warden imports PATH only, not `$AGENTMUX_*` or other vars, so a probe that depends on a shell export still sees it empty under a GUI launch. The canonical probe sidesteps all the socket detail by delegating to **`"$HOME/.agentmux/bin/amux" --probe`**: amux owns the agent/frame socket layout (agent on tmux's default socket via `env -u TMUX tmux`; frame on the hard-coded `-L agentmux-frame`), so warden config carries none of it and can't drift from amux. Two things warden must still get right: name **`amux` by absolute path** (a Finder launch's minimal PATH won't find a bare `amux`), and rely on the login-PATH import for amux's own `tmux` lookups. An absolute-path binary remains the robust fallback in general (covers a login-PATH derivation that fails/times out, or a PATH set only in interactive-only rc that `-l` doesn't source). The notification path is immune (it flows tty→tmux passthrough→libghostty, never through warden's PATH).
- **macOS banners go through the native `UNUserNotificationCenter`, not `tauri-plugin-notification` (the "notifications silently do nothing on macOS 26" bug):** the Tauri plugin's desktop backend is `notify-rust` → `mac-notification-sys`, which posts via the **deprecated `NSUserNotification` API**. On macOS 26 that API is a silent no-op — `show()` returns `Ok`, but nothing is delivered and the app never even registers in Notification Center (`com.apple.ncprefs`). The plugin's `request_permission`/`permission_state` are also no-op stubs that always return `Granted`, so you can't fix it by requesting permission. warden therefore posts banners itself via the UserNotifications framework (`notify.rs`, objc2 `UNUserNotificationCenter`). Three things that path **requires**, all handled in `setup_banners`: (1) request authorization once at startup (the one-time macOS prompt); (2) install a `UNUserNotificationCenterDelegate` whose `willPresentNotification` returns `.banner` — otherwise the system suppresses banners while warden is the *frontmost* app, which is exactly warden's hidden-tab-in-focused-window case; (3) guard on `!tauri::is_dev()` — `currentNotificationCenter` throws on a nil bundle identifier, so native banners only run from the packaged `warden.app` (dev still badges). Don't reintroduce the plugin for banners.
- **"No notifications at all" (not even the amber badge) is a *transport* failure upstream of warden, not a warden bug.** warden only badges/banners when the OSC 9/777 escape actually reaches its libghostty surface; if neither the badge nor the banner appears, the escape never arrived — look at the *producer*, not `notify.rs`. The canonical producer (agentmux's `--notify`) wraps the OSC in a tmux passthrough envelope, and **a single wrap only escapes one tmux layer**: running an agent under `amux --frame` puts it two tmux deep (frame → agent), so a single-wrapped escape is unwrapped by the inner tmux and dropped by the frame before warden sees it. The producer must wrap once per nested tmux layer (fixed in agentmux ≥ 0.14.0). Diagnose by capturing the agent→frame boundary: `tmux -L agentmux-frame pipe-pane -t <agent-pane> 'cat >> /tmp/cap'` — a bare `]777` crossing means it's dying at the frame; a `Ptmux;` envelope means it'll reach warden. (The badge is independent of macOS auth, so badge-but-no-banner is the *auth/`is_dev`* case above or the *macOS-26 presentation* case below — not the transport case here.)

- **A missing banner when the badge *does* show, on macOS 26, is the OS — not `notify.rs` (the "the banner code must be broken" trap).** The whole warden side of the banner path was verified on a notarized build: the OSC decodes to a `Notification`, the tab is correctly seen as not-visible, `show_banner` posts, **usernoted accepts the request**, and the delegate's `willPresentNotification` **fires returning `.banner | .sound`** — with `getNotificationSettings` reporting Authorized/Banner/Enabled and no Focus assertion active. Everything the API requires succeeds and the banner still doesn't render, because macOS 26 has a known notification-presentation regression (corrupted background service; both `au.lsjc.warden` and sibling `au.lsjc.curator` were absent from System Settings → Notifications despite being authorized). **The regression has a foreground-only variant:** on one affected machine banners rendered fine when warden was *backgrounded* and dropped only when a warden window was *frontmost* — the trace still showing `willPresent fired -> returning .banner | .sound` (macOS consults `willPresent` only in the foreground, so the fault lands only on that path). So don't let warden's own backgrounded banners working exonerate the foreground path — the discriminator is "do *foreground* banners appear from *any* app (bring curator/Messages frontmost and fire one)?", not "do any banners appear?". So a missing banner here is fixed by the **OS** (reboot — `killall ControlCenter NotificationCenter` without a re-login was confirmed *not* to fix it — then the "mirroring/sharing" gate, notification-DB clear, or Safe Mode), not by editing the notification code. Set `notify_debug = true` (config) to re-capture the trace to `/tmp/warden-notify-dbg.log` after a reboot. Full write-up + fixes: **docs/notifications.md**. Don't "degenericise" or rewrite `notify.rs` chasing this — the trap is that every readable signal says "should show a banner."
- **The empty-state placeholder is composited BEHIND the native surface — the inverse of curator's placeholder webview:** the "blank background when no tab is selected" (`#empty-state` in `index.html`) is a page-shell element *inside* `#terminal-hole`, shown by the controller (`paintEmptyState`) only when no tab is selected — it stays in warden's shell, NOT in chrome-core (chrome-core is the sidebar view; the empty-state is a content-area concern). It works because warden's terminal NSView composites *above* the WKWebView, so a live tab's surface naturally occludes it and it shows through only when the hole has no surface. This is **opposite** to curator, which stacks a separate `placeholder.html` *webview behind* its content webviews (curator's content is webviews, which it z-orders; warden's content is a native surface that's always on top). So do **not** "port" curator's separate-webview placeholder mechanism here — there's no webview to stack, and a separate webview would render *over* the chrome, not under the terminal. It must stay **opaque** (the window is transparent; a non-opaque fill leaks the wallpaper — the exact leak it exists to stop), and the toggle stays gated on `active` so it can't bleed through a translucent terminal theme. The shield SVG is hand-recoloured from `assets/icon-app.svg`'s path geometry (muted grey, no tile/gradient) — if the app mark changes, re-derive it from the same source.
- **Keep the migration complete:** this is a young codebase; finish changes in-branch, no transitional fallbacks.
- **Stale frontend embed:** `tauri-build` (2.6.x) emits no `rerun-if-changed` for `frontendDist`, and the `ui/` assets are embedded at compile time via `generate_context!`. So after a *frontend-only* edit (`index.html`/`diagnostic.html`), `cargo run` silently serves the **stale** HTML until some Rust change forces a recompile. `crates/warden-app/build.rs` watches the **hand-written** assets (`cargo:rerun-if-changed=ui/index.html` + `ui/diagnostic.html`) to defeat this — keep them. It deliberately does **not** watch `ui/` broadly: build.rs itself writes the generated `ui/chrome-core.{css,js}` (materialized from the pinned chrome-core crate) into that dir, so a broad watch would self-trigger a rerun every build. If you edit the frontend and the change "doesn't take," suspect a stale embed (touch a `.rs` file or `cargo clean -p warden-app`).

## Relationship to curator (the shared chrome-core component)

warden and **curator** ([github.com/Lockyc/curator](https://github.com/Lockyc/curator)) — a sibling public macOS Tauri app that curates **browser keeper-tabs** (webviews) where warden curates **terminals** (libghostty surfaces) — share the same sidebar chrome, extracted into **chrome-core** ([github.com/Lockyc/chrome-core](https://github.com/Lockyc/chrome-core)). Both apps consume it, so a look/behaviour change is made once and both move together.

- **The sidebar chrome IS chrome-core (the view) — change it there, not here.** chrome-core owns the banner + accent tint, grouped tab rows (letter tile, title, the three dot slots: attention / presence / live-unload), the kill-confirm row overlay, density tokens, the resize-drag, and the error bar. It's a build-dependency pinned by `rev`; `build.rs` writes `SIDEBAR_CSS`/`SIDEBAR_JS` into `ui/chrome-core.{css,js}` (git-ignored) before `generate_context!` embeds them. `ui/index.html` is now a **page shell + thin controller**: the `#app`/`#terminal-hole`/empty-state topology, `reportRect`, the `forMe` per-window filter, the first-probe handshake, and a controller that maps the component's callbacks (`onSelect`→`activate_tab`, `onUnload`→`unload_tab`, `onKill`→`kill_session`, `onResize`→CSS width) and events (`warden:refresh`→`update`, `warden:notify`→`setAttention`, `warden:session-state`→`setPresence`, `warden:error*`→`setError`/`clearError`) to warden's backend.
- **What stays per-app is topology + backend, not the sidebar.** warden keeps its native-surface hole (`#terminal-hole` + `reportRect` + the empty-state placeholder) and its Tauri commands/events; curator keeps its content webviews + its own commands. These are the ~40-line controllers, not chrome duplication. warden feeds the component a normalized tab DTO where `presence` (probe) and `killable` (kill) are warden-only — curator never sets them — and `attention` carries no count (curator's unread count does). warden passes **no** `active` field, so the component owns selection (preserve + fall back to first, firing `onSelect` to activate); curator passes `active` because its Rust side is authoritative.
- **Dev iteration on the chrome behind a pinned rev is higher-friction.** Use a cargo path override (`[patch."https://github.com/Lockyc/chrome-core"]` → a local checkout) while actively working the chrome, then re-pin the rev before committing. Edit chrome-core's `assets/sidebar.{css,js}` — never the generated `ui/chrome-core.*`.
- **Terminal-only vs browser-only affordances are DTO/callback differences, not ports.** A terminal has session state, so warden's unload *kills* the terminal (no cheap re-fetch, no curator-style reload-on-click). curator's browser nav (back/forward/home/reload) mounts into the component's optional `header` slot, which warden leaves null.
